static linking

static linker有两个任务： - symbol resolution:主要是函数，全局变量，静态变量；目的是确保每个符号只有一个定义 - relocation

obj文件仅仅只是一些block的集合。一些block是code，一些是data，一些是指导loader和linker工作的数据结构block。 # object files obj文件有三种形式： - relocatable obj（由compiler和assembler产生） - executable obj （linker产生） - shared obj （由compiler和assembler产生）

通过file命令可以知道ELF文件是上面的哪个类型

symbols and symbols tables

每一个relocatable module有三种symbol：自己定义的可以被其他module引用的global symbols；引用其他module的global module；自己定义和引用的local symbols。

所以.symtab中没有local nonstatic program variables。

local static variables不在栈中管理。而是在.data和.bss中定义local linker symbols.

dynamic linking with shared lib

static lib的缺点 - 库更新不方便。 - 有些通用的函数如printf每个程序一份也很浪费资源。

我们先部分链接再全部链接的方式。在最后的可执行文件prog21会包含.interp指定了动态链接器（在Linux上是ld-linux.so，自己本身也是一个shared object）的路径。然后先load这个动态链接器，完成relocate，再把控制交给程序本身。

loading and linking shared libs from app

不仅程序再执行前可以加载和链接shared object，运行中也可以！且不需要编译时候的链接。

困扰我很久的opengl的加载函数的问题是不是就解决了？

有四个接口dlopen,dlsym,dlclose,dlerror。 然后用gcc -rdynamic -o prog2r dll.c -ldl来编译下面的文件！

pic位置无关代码

可执行obj模块内部的符号引用只需要pc-relative然后通过static linker就可以解决。pic的目的是为了解决stared module的external procedure和global variables的引用问题。
